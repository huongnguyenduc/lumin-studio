# Filament-sourced colors + per-part 3D color ‚Äî feature plan

> **House style** = `phase-4-extension.md` / `p3l-product-editor.md`: owner decisions locked up front ‚Üí dependency-ordered, merge-gated slices, each = one user-merge-gate PR with a one-line "done-when". Slices prefixed `f-` (filament-color). **[MONEY]** = touches pricing/capture (ADR-019 + ADR-037 immutable snapshot) ‚Üí extra care.
>
> **Source:** owner ask 2026-07-16 (per-part exact color + "product colors come from filaments, not a separately-typed name/hex"). Grounded by an `oracle` design pass + a `Plan` surface sweep. **ADRs are NOT written yet** ‚Äî they land via the `.allow-contract-edit` valve at implementation time, with the owner's explicit opt-in ([[contract-edit-deliberate]]).

## 0 ¬∑ One-line
A product **part** offers **filaments**; a color's display name + hex **come from the linked `filament_materials` row** (no typed name/hex). Each named part then renders in its exact color in 3D ‚Äî the **static sprite** in the part's *default* filament color, and the storefront **live viewer** recoloring per-part in real time as the customer picks. Feasible because the printable model is one file with **each part a named object/material** (owner-confirmed).

## 1 ¬∑ The finding that de-risks the epic
**The filament plumbing already exists end-to-end** ‚Äî `colors.filament_material_id` (migration `000019:34`, added for deduct-on-print), `Color.filamentMaterialId` in openapi (`:2872`/`:3528`), and the admin color dialog **already renders a filament `<Select>`** (`product-colors-dialog.tsx:338`) *alongside* today's name/hex inputs. So "colors from filaments" (scope 1) is mostly a **read-through refactor**: remove the name/hex inputs, make the filament required, resolve name/hex from the filament. The genuinely-new build is the **3D per-part color** (sprite + live viewer) plus the **part‚Üîobject-name mapping** both consumers share.

Also verified on-box today (o-1c): Blender sees the GTX 1060 CUDA device, renders a 24-frame sprite in ~1:55, and **renders a model's own material color** (a red-material GLB ‚Üí red frame). The render infra for the sprite slice is proven.

## 2 ¬∑ Locked decisions (oracle-resolved; where oracle and Plan diverged, oracle wins)

| # | Decision | Resolution | Why / ceiling |
|---|---|---|---|
| **D-A** | How name/hex "read through" from filament | **Copy-on-write.** Keep `colors.name/hex` columns; `InsertColor`/`UpdateColor` **stamp** name+hex from the linked filament at write time. **ADR classification** (written at implementation time, valve + opt-in):
- **New (small) ADR** ‚Äî part‚Üíobject mapping (reverses the ADR-037/038 "no geometry mapping" clause; adds only a name handle, still no zones/coords).
- **New ADR** ‚Äî second structured glb + `model3d_structured_url` (changes the ingest output contract; reverses ADR-038 "worker doesn't change"). *Live-viewer track only.*
- **Amend ADR-049** ‚Äî payload grows the `partColors` map (sprite).
- **Amend ADR-039** ‚Äî `filament_material_id` extends from "costing-only" to "costing + swatch source" (copy-on-write).
- **Within ADR-049** (no ADR) ‚Äî first-color default + manual re-render (reuses `CreateProductAssetJob`, `admin_asset_jobs.go:88`; there is no auto-enqueue-on-catalog-change today and we are not adding one).

## 5 ¬∑ Risks + lazy-v1 de-scope
- **Smallest shippable core = f-1 alone.** "Color comes from filament" ships with zero 3D work. If the epic must be cut, f-1 is the value.
- **After f-1/f-2, the two 3D tracks are independent ‚Äî pick order by appetite:**
  - **Sprite-first (f-5):** the literal original ask; **lowest risk** (no ingest change); **reuses the o-1c render proven on-box today**. Downside: box-gated verification, and value is a default color on a card thumbnail.
  - **Viewer-first (f-4‚Üíf-3):** **higher value** (interactive "pick and see" ‚Äî the real made-to-order experience) and off-box verifiable. Downside: needs the riskier ingest second-artifact + one unverified assumption (below).
- **Biggest risks (oracle):**
  1. **Un-fusing regresses normalization.** Mitigated by D-B: never touch the fused output; emit a second artifact with the *same* translation; keep Track B behind its own gate.
  2. **Silent color-detach from mapping drift.** Mitigated by the explicit ingest-sourced dropdown + NULL/unmatched ‚Üí default filament (not grey) + a visible "part not mapped / sprite stale" signal in the editor.
  3. **Hex crossing the render trust boundary + idempotency.** Mitigated by server-side `#RRGGBB` validation before the payload + denormalized frozen snapshot (never query-back) + update the `job.rs` wire test in the same PR.
  4. ~~**(Track B, unverified from code)** that `<model-viewer>` can address a named material and `setBaseColorFactor` at runtime through `export_glb`~~ ‚Üí **‚úÖ RESOLVED (f-3)**: proven in a real browser (Chrome automation) ‚Äî a 2-object structured glb loaded in model-viewer 4.3.1; `getMaterialByName("Chao ƒë√®n"/"ƒê·∫ø")` resolved by object name; `setBaseColorFactor` (hex-string AND RGBA-array) changed the rendered pixels red‚Üígreen+blue. Gated on an UNCOMPRESSED glb (ingest emits no Draco/KTX2 ‚Üí zero gstatic/PDPL fetch, holds). The one addition it required: `ingest.py` names materials = objects (model-viewer keys recolor by material, mesh‚Üímaterial is symbol-private).
