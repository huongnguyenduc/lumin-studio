# Filament-sourced colors + per-part 3D color — feature plan

> **House style** = `phase-4-extension.md` / `p3l-product-editor.md`: owner decisions locked up front → dependency-ordered, merge-gated slices, each = one user-merge-gate PR with a one-line "done-when". Slices prefixed `f-` (filament-color). **[MONEY]** = touches pricing/capture (ADR-019 + ADR-037 immutable snapshot) → extra care.
>
> **Source:** owner ask 2026-07-16 (per-part exact color + "product colors come from filaments, not a separately-typed name/hex"). Grounded by an `oracle` design pass + a `Plan` surface sweep. **ADRs are NOT written yet** — they land via the `.allow-contract-edit` valve at implementation time, with the owner's explicit opt-in ([[contract-edit-deliberate]]).

## 0 · One-line
A product **part** offers **filaments**; a color's display name + hex **come from the linked `filament_materials` row** (no typed name/hex). Each named part then renders in its exact color in 3D — the **static sprite** in the part's *default* filament color, and the storefront **live viewer** recoloring per-part in real time as the customer picks. Feasible because the printable model is one file with **each part a named object/material** (owner-confirmed).

## 1 · The finding that de-risks the epic
**The filament plumbing already exists end-to-end** — `colors.filament_material_id` (migration `000019:34`, added for deduct-on-print), `Color.filamentMaterialId` in openapi (`:2872`/`:3528`), and the admin color dialog **already renders a filament `<Select>`** (`product-colors-dialog.tsx:338`) *alongside* today's name/hex inputs. So "colors from filaments" (scope 1) is mostly a **read-through refactor**: remove the name/hex inputs, make the filament required, resolve name/hex from the filament. The genuinely-new build is the **3D per-part color** (sprite + live viewer) plus the **part↔object-name mapping** both consumers share.

Also verified on-box today (o-1c): Blender sees the GTX 1060 CUDA device, renders a 24-frame sprite in ~1:55, and **renders a model's own material color** (a red-material GLB → red frame). The render infra for the sprite slice is proven.

## 2 · Locked decisions (oracle-resolved; where oracle and Plan diverged, oracle wins)

| # | Decision | Resolution | Why / ceiling |
|---|---|---|---|
| **D-A** | How name/hex "read through" from filament | **Copy-on-write.** Keep `colors.name/hex` columns; `InsertColor`/`UpdateColor` **stamp** name+hex from the linked filament at write time. Admin removes the name/hex inputs; filament picker becomes **required for part-colors**. | Smallest diff — every reader (`colorsDTO`, `colorDTO`, capture) stays byte-identical (still reads `c.Name`/`c.Hex`). No `colors` reshape, no nullable migration. `ponytail:` a filament rename won't retro-update saved colors until re-saved — acceptable; rename-cascade is the upgrade path. |
| **D-A′** | NULL-hex filaments | **Only filaments *with* a hex are selectable as a color source.** | Makes the bad state (color with no swatch) unrepresentable instead of NULL-checked across 4 surfaces + 2 render paths (`filament_materials.hex` is nullable — "no colour chip", `000018:14`). |
| **D-B** | Does `model_ingest` stop fusing? | **NO.** Keep the fused `model.glb` **unchanged**; emit a **SECOND** `model_structured.glb` (named objects/materials preserved, **same** recenter translation), new column `products.model3d_structured_url` (callback-written, single-writer like `model3d_url`). Viewer uses structured when present, falls back to `model3d_url`. | oracle: `force='mesh'` is load-bearing — it makes dims-prefill / recenter / watertight well-defined and is what ADR-038's camera pose is curated against. Un-fusing regresses that. **Only needed for the live-viewer track.** Landmine: both exports must apply the *same* translation vector or the curated pose de-frames one. |
| **D-C** | Default (sprite) color per part | **First `available` color of the part by `display_order` → its filament hex.** No new column. | Reuses the storefront's existing first-color-preselect convention. `ponytail:` add an explicit `default_filament` picker only if a shop wants a default ≠ first swatch. Ceiling: sprite shows one fixed colorway (already ADR-049 reality — card's primary image is the shop photo). |
| **D-D** | How the live viewer gets filament color **publicly** (filament is admin-only today) | **Reuse public `Color.hex`/`name`** (free once read-through lands) **+ add `Part.modelObjectName`** to the public product. No new public filament endpoint. | Don't leak internal filament fields (cost/stock) onto the public `Product`. |
| **D-E** | How per-part colors reach the sprite worker | **Denormalize** a `{objectName → hex}` map into the `asset_job.created` payload. **Validate hex `#RRGGBB` server-side** before it enters the payload; update the `job.rs` wire-contract test. | oracle: this is *more* ADR-006-compliant than a worker→core-api query — the job stays a frozen, reconstructable snapshot, so re-running an old job can't produce a different sprite from live catalog (idempotent). A malformed hex = rejected poison job, never a grey fallback that ships wrong. |
| **D-F** | Legacy flat colors (`part_id` NULL, no filament) | **Keep working unlinked.** filament required only for **part**-colors; flat colors keep their own name/hex. | Preserves ADR-037 backward-compat + immutable capture. |
| **Map** | Part↔object mapping | `parts.model_object_name text` **nullable**. Admin assigns each part **two ways** (owner ask 2026-07-16): (1) a **dropdown auto-suggested by name-match** on `parts.name` (`ingest.py` extracts the object/material name list, ~3 lines `scene.geometry.keys()`, no un-fuse), **and** (2) **click the mesh in a 3D preview** — reuse the ADR-038 admin viewer `product-model-view.tsx` + model-viewer hit-test (`surfaceFromPoint`/material-from-point → object/material name → assign to the selected part). NULL/unmatched → part renders in its **default filament**, never grey. | oracle: pure name-match convention detaches colors *silently* on a rename / casing / diacritic drift ("Chao đèn" vs "Chao Đèn") — too low a ceiling for an owner-edited, money-adjacent field; click-on-model removes the typing entirely for the happy path. **Named-model source for the click viewer** (f-2 design sub-decision): prefer the retained **source glb** (`sourceModelUrl` from the asset-job DTO, `admin_asset_jobs.go:170`) since it keeps its object names — if it isn't browser-reachable/glb, fall back to f-4's structured derivative (→ sequence f-4 before f-2's click UI). `ponytail:` one object-name per part in v1 (`text[]` widening later); click-to-pick degrades to dropdown for nameless STL sources. |

**ADR classification** (written at implementation time, valve + opt-in):
- **New (small) ADR** — part→object mapping (reverses the ADR-037/038 "no geometry mapping" clause; adds only a name handle, still no zones/coords).
- **New ADR** — second structured glb + `model3d_structured_url` (changes the ingest output contract; reverses ADR-038 "worker doesn't change"). *Live-viewer track only.*
- **Amend ADR-049** — payload grows the `partColors` map (sprite).
- **Amend ADR-039** — `filament_material_id` extends from "costing-only" to "costing + swatch source" (copy-on-write).
- **Within ADR-049** (no ADR) — first-color default + manual re-render (reuses `CreateProductAssetJob`, `admin_asset_jobs.go:88`; there is no auto-enqueue-on-catalog-change today and we are not adding one).

## 5 · Risks + lazy-v1 de-scope
- **Smallest shippable core = f-1 alone.** "Color comes from filament" ships with zero 3D work. If the epic must be cut, f-1 is the value.
- **After f-1/f-2, the two 3D tracks are independent — pick order by appetite:**
  - **Sprite-first (f-5):** the literal original ask; **lowest risk** (no ingest change); **reuses the o-1c render proven on-box today**. Downside: box-gated verification, and value is a default color on a card thumbnail.
  - **Viewer-first (f-4→f-3):** **higher value** (interactive "pick and see" — the real made-to-order experience) and off-box verifiable. Downside: needs the riskier ingest second-artifact + one unverified assumption (below).
- **Biggest risks (oracle):**
  1. **Un-fusing regresses normalization.** Mitigated by D-B: never touch the fused output; emit a second artifact with the *same* translation; keep Track B behind its own gate.
  2. **Silent color-detach from mapping drift.** Mitigated by the explicit ingest-sourced dropdown + NULL/unmatched → default filament (not grey) + a visible "part not mapped / sprite stale" signal in the editor.
  3. **Hex crossing the render trust boundary + idempotency.** Mitigated by server-side `#RRGGBB` validation before the payload + denormalized frozen snapshot (never query-back) + update the `job.rs` wire test in the same PR.
  4. **(Track B, unverified from code)** that `<model-viewer>` can address a named material and `setBaseColorFactor` at runtime through `export_glb` — **verify material-preservation-through-export on a real 2-part model before committing f-3** (also gated on an uncompressed glb; Draco/KTX2 would reintroduce the gstatic-WASM/PDPL problem flagged at `model-3d-viewer.tsx:68`).
