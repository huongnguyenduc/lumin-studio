### ADR-001 — Monorepo pnpm + Turborepo · Accepted
4 bề mặt + 1 state machine + 1 design system ⇒ cần lõi chia sẻ (`packages/core`, `tokens`, `ui`, `api-client`). Turborepo cache + build affected-only. **Loại:** 4 repo rời (đồng bộ state machine khổ).

### ADR-002 — Frontend Next.js (storefront, admin) + MV3 (extension) · Accepted
Next.js: SSR/SEO cho catalog, mobile-first, hệ sinh thái lớn, dễ tích hợp model-viewer. Admin Mobile = **cùng app admin responsive**, không tách codebase. **Loại:** Vue/Nuxt (đội không có lợi thế rõ).

### ADR-003 — Core API: Go + Chi v5 (+ hợp đồng OpenAPI) · Accepted
Chi là pure `net/http` → toàn hệ sinh thái dùng được; binary tĩnh idle ~15–40MB, hợp box nhà (tải thật nằm ở Postgres/NATS/Garage/Blender). Toolchain: **chi v5 + pgx v5 + sqlc + go-chi/jwtauth + RBAC nhỏ**. **Bắt buộc lớp hợp đồng** vì Chi không cho type TS: **OpenAPI → oapi-codegen (Go) + openapi-typescript/openapi-fetch (3 client TS)**. **Loại:** Node/NestJS (idle ~80–150MB+, đã rời bỏ); Fiber (fasthttp, mất tương thích net/http); Gin/Echo (context riêng). Connect-RPC để dành cho surface streaming/admin nếu sau này cần.

### ADR-004 — PostgreSQL (app) + Postgres DB riêng cho Umami · Accepted
JSON column cho statusHistory/address/personalization. Truy cập qua sqlc+pgx (type-safe, nhanh hơn GORM). Umami để **DB riêng** tránh query nặng khoá bảng app.

### ADR-005 — Object storage: Garage, all-home (KHÔNG MinIO, KHÔNG R2) · Accepted
**MinIO CE đã archive (2/2026)**, gỡ admin console, ngừng publish Docker image từ 10/2025 → không xây mới trên đó. Garage: S3-compatible, ~1GB RAM, single-node, còn maintain. User chọn **all-home** (không R2). **Hệ quả bắt buộc:** Garage **không có object versioning** → backup ngoài là lưới an toàn duy nhất; serve .glb/sprite qua **Cloudflare cache** với URL **content-hash bất biến + TTL dài** (giữ asset nhỏ); upload model qua **S3 multipart presigned-PUT, part <100MB** (Cloudflare Tunnel chặn body >100MB). Lối thoát nếu vướng ToS file lớn của CF: bật bucket R2 cho asset public (cùng API S3, không sửa code).

### ADR-006 — Queue: NATS JetStream + outbox (KHÔNG River) · Accepted
Worker là **Rust** (ADR-007) → River (Go-only, crate Rust mới 0.0.0) không dùng được. NATS JetStream: broker language-agnostic nhẹ nhất cho 1 box, ack-wait dài + heartbeat cho render lâu, MaxDeliver + stream DLQ. **Bắt buộc outbox / publish-on-commit** (publish job chỉ sau khi row AssetJob commit) để tránh mất job do dual-write. **Lưu ý:** nếu sau này worker chuyển sang Go, cân nhắc River để bỏ hẳn broker.

### ADR-007 — Asset Worker: Rust + Blender, GIỮ render 360° server-side · Accepted (user override)
Nghiên cứu khuyến nghị worker Go + bỏ turntable (model-viewer client-side đã lo xoay 360°), nhưng **user chọn giữ** để fidelity cao + làm fallback no-WebGL. **Ràng buộc bắt buộc:** render bằng **Cycles + CUDA** (KHÔNG OptiX — Pascal không RT core, phí VRAM; KHÔNG EEVEE — chết headless); OpenImageDenoise chạy **CPU** trên CC 6.1; **subprocess** Blender (không bpy) để crash-isolation + retry; **concurrency=1**, off-peak; decimate + ≤1080p + sample vừa để vừa 6GB VRAM. WSL2 GPU: chỉ cài driver NVIDIA trên Windows + cuda-toolkit trong WSL2, **không** cài driver Linux trong WSL2; validate Blender thấy GPU trong container (Blender #126014) trước khi coi là xong. ⇒ **32GB RAM gần như bắt buộc**. **Worker KHÔNG render poster** — ảnh đại diện/list của SP dùng **ảnh shop chụp** (`Product.images[0]`); 360° sprite chỉ dùng cho hover (PC)/dừng-2s (mobile) trên card + fallback no-WebGL của viewer 3D.

### ADR-008 — Realtime: SSE từ Core API · Accepted
Tiến độ là server→browser thuần → SSE (EventSource auto-reconnect). NATS **không** lộ ra browser. **Bắt buộc** cấu hình chống buffer của cloudflared (no-transform, Content-Encoding: identity, X-Accel-Buffering: no, http.Flusher mỗi event, heartbeat) + **smoke-test qua named tunnel**; fallback = polling ngắn (admin ít người, thừa sức). **Loại:** NATS-over-WebSocket ra browser (lộ broker).

### ADR-010 — Thanh toán: đối soát VietQR **thủ công** (SePay/Casso bị từ chối) · Accepted (user)
User giữ thủ công. **Luồng web (pay-then-confirm):** checkout **chưa tạo đơn** → màn **QR tĩnh** (render server-side từ STK đã lưu; **nội dung/memo CK không bắt buộc**) → khách **gửi ảnh chụp biên lai + bấm xác nhận** thì frontend mới `POST /orders` tạo đơn ở `PENDING_CONFIRM` (kèm ảnh CK). Vì khoảng chờ "đã CK → xác nhận" là rủi ro bỏ giỏ lớn nhất, các bù trừ thành **bắt buộc**: màn "chờ xác nhận" (khách vào qua **link tra cứu đơn**) có timeline + thời gian dự kiến + **auto-poll**; shop **đối soát bằng ảnh biên lai** khách gửi (đối chiếu số tiền) → **nút 1-chạm → PAID** (owner-only) trên Admin + Admin Mobile; email link+mã đơn. SePay/Casso (webhook real-time) là option để dành — nếu bật sau này cần thêm memo/định-danh-đơn để auto-match (QR tĩnh hiện không mang). **Loại (đừng làm):** tự scrape ngân hàng/parse email; cổng VNPay/MoMo/ZaloPay (giai đoạn 1 chỉ bank_transfer).

### ADR-012 — Hàng cá nhân hoá: cọc + trả trước + no-COD + huỷ/hoàn-tiền · Accepted (user)
Hàng khắc không bán lại được → COD bị bom là mất trắng. **Đặt cọc (Điều 328 BLDS) + mặc định trả đủ trước qua VietQR + không COD hàng khắc + tickbox "không đổi trả hàng cá nhân hoá" trước thanh toán** (vô hiệu hoá quyền huỷ 30 ngày của Luật BVNTD 19/2023; vẫn giữ nghĩa vụ in lại nếu lỗi/sai). Bước **echo lại nội dung khắc** trước khi trả tiền.

**Cập nhật 2026-06-24 (user) — mô hình huỷ/hoàn tiền (đồng bộ `spec.md §04`):**
- **Bỏ `RETURNED`.** Trả đủ 100% trước + hàng custom không tái sử dụng ⇒ không nhận hàng trả về. Mở lại chỉ khi sau này có mặt hàng không-cá-nhân-hoá có thể trả ("chuyện rất lâu").
- **Thêm state `REFUNDED`** thay vì gắn metadata refund vào `CANCELLED`. `CANCELLED` = đóng đơn **không** hoàn tiền (tiền chưa về, hoặc lỗi khách + đã in → giữ tiền); `REFUNDED` = **đã chuyển hoàn** (thường lỗi shop), kèm `refundProofUrl`. "Fault do shop/khách" biểu đạt bằng chính lựa chọn `CANCELLED` vs `REFUNDED` — không cần field riêng. Cả hai terminal; không có đường ra khỏi `COMPLETED`.
- **Inbox vào thẳng `PAID`:** nhân viên tự kiểm tra thấy tiền về rồi mới tạo đơn ⇒ không qua `PENDING_CONFIRM` (state này chỉ còn ở kênh web, nơi chủ đối soát ảnh biên lai).
- **RBAC tiền-vào vs tiền-ra:** tạo đơn inbox-đã-`PAID` cho phép **nhân viên** (họ là người verify tiền, rủi ro thấp); nhưng `→ REFUNDED` và sửa `bankAccount`/STK **owner-only** (tiền-ra + config khoá cho chủ). Bổ sung cho ADR-010 (`PENDING_CONFIRM → PAID` web vẫn owner-only).
- **Kế toán:** doanh thu ròng = đã thu − `REFUNDED`; `CANCELLED` sau `PAID` không hoàn = shop giữ tiền (vẫn là doanh thu). Hàng custom không nhập lại kho; chỉ ghi filament đã tiêu thành phế phẩm (Spoolman) nếu huỷ sau PRINTING.
- Copy trấn an checkout bỏ "đổi trả miễn phí 30 ngày" → "in lại miễn phí nếu lỗi do shop"; `Setting.returnPolicy` → `refundPolicy`.

### ADR-021 — Kiến trúc điều khiển Claude Code (harness): advisory→deterministic · Accepted
Để hạ tỉ lệ sai/lệch: CLAUDE.md/rules là *advisory* (không đảm bảo tuân thủ — model có thể bỏ qua), nên mọi luật "luôn-phải" được đẩy xuống tầng *deterministic*. Triển khai trong `.claude/` (commit vào repo): **hooks** — `guard-bash` (chặn rm -rf, `docker compose down -v`, git push --force, mkfs/dd, DROP/TRUNCATE), `guard-files` (chặn cứng secrets; **ask** khi sửa file hợp đồng), `format-and-lint` (PostToolUse Prettier+ESLint per-file), `verify-before-stop` (Stop hook chặn dừng tới khi typecheck/lint/test xanh, check `stop_hook_active`, van xả `.claude/.skip-verify`); **`.claude/rules/*.md`** path-scoped (load theo bề mặt đang chạm, giảm context rot); **`agents/spec-guardian`** (reviewer ngữ-cảnh-mới soi diff theo ADR/conventions). Mọi hook **tự no-op** tới khi tool/script tồn tại (an toàn trước Phase 0) và gọi script trừu tượng nên độc lập version. Kỷ luật quy trình: plan-to-file cho việc nhiều-file, `/clear` sau 2 lần sửa sai, single-thread + subagent kiểm chứng cho code (multi-agent chỉ cho nghiên cứu đọc rộng). **Loại:** chỉ dựa CLAUDE.md (advisory, dễ bị bỏ qua — xem issue #21119); over-fragment nhiều agent song song cho việc viết code (Cognition "Don't Build Multi-Agents"). Chi tiết: `agent-harness.md`. **(Mệnh đề "ask khi sửa file hợp đồng" được ADR-022 siết lên hard-block cho `decisions.md`+`conventions.md`; phần còn lại giữ nguyên. Mở rộng harness vòng 2: ADR-023.)**

### ADR-022 — Siết file hợp đồng lõi (decisions/conventions) từ ASK lên hard-block · Accepted (supersedes một phần ADR-021)
Từ vòng nghiên cứu harness (2026-06-23, run wf_1a2ff429; chi tiết `agent-harness-learnings.md`). ADR-021 đặt `guard-files` ở mức **ask** cho mọi file hợp đồng. ask là gate **mềm** — một "yes" lỡ tay giữa phiên có thể đổi luật binding mà cả harness đang ép theo (Aider tách `/read` chỉ-đọc khỏi `/add`; thứ tự quyền của Claude Code: **deny > ask**). **Quyết định:** `guard-files.sh` **hard-block (exit 2)** với `docs/decisions.md` + `docs/conventions.md`; **van xả** là file `.claude/.allow-contract-edit` (mirror `.skip-verify`) — khi tồn tại thì rơi xuống mức ask cũ. **Giữ nguyên ask** cho `tokens/*.css` + `CLAUDE.md` + `AGENTS.md` (Phase 0 sửa thường xuyên). **Đường amend chính thức** cho mọi thay đổi ADR/luật (kể cả viết ADR mới): `touch .claude/.allow-contract-edit` → sửa (chỉ THÊM ADR / đánh dấu Superseded) → `rm .claude/.allow-contract-edit`. **Phạm vi supersede:** chỉ mệnh đề "ask file hợp đồng" của ADR-021 cho 2 file lõi. **Loại:** hard-block luôn cả tokens/CLAUDE.md (ma sát Phase 0); chỉ dựa prompt/spec-guardian (advisory, không đảm bảo).

### ADR-023 — Mở rộng harness vòng 2: orient / anti-reward-hacking / anti-loop / skills / oracle / self-test · Accepted
Từ nghiên cứu các harness coding-agent phổ biến (Claude Code lifecycle hooks, Cline/Roo Memory Bank, OpenHands StuckDetector, Amp Oracle, SWE-agent ACI, spec-kit/Kiro, AGENTS.md) — bảng đối chiếu "đã có vs còn thiếu" + lý do từng khuyến nghị ở **`agent-harness-learnings.md`**. **Giữ nguyên** nguyên lý advisory→deterministic (ADR-021) và **không** đổi quyết định cũ; chỉ bổ sung: (1) **SessionStart hook** `session-start.sh` — orient (nhánh + index plan + `active-context.md`) + reset state loop/retry. (2) **Anti-reward-hacking** — `guard-files` ask khi test bị làm yếu (thêm `.skip`/`t.Skip`, giảm assertion) + spec-guardian coi xoá test/bỏ assertion trên invariant lõi là **BLOCKER**. (3) **Anti-loop** — retry budget trong `verify-before-stop` (4 lần fail cùng target → surface cho người, không loop vô hạn — vd test GPU không pass nổi trên GTX 1060) + loop-detector lệnh y hệt (≥4×) trong `guard-bash`. (4) **Parse-gate Go** (`gofmt -e`) sớm trong `format-and-lint`. (5) **Tier `.claude/skills/`** (vn-compliance, render-worker-gpu, event-outbox — pointer mỏng, **defer** về doc nguồn). (6) **Agent `oracle`** (design second-opinion, read-only, gọi tay, **không bao giờ** là gate). (7) **Harness self-test** `tests/harness/guard.test.sh` chạy ở CI khi `.claude/**` đổi (gate no-op = rủi ro vô hình). (8) **`AGENTS.md`** pointer cross-tool mỏng. (9) **Acceptance ledger** `docs/acceptance.md` (EARS, test-linked) + **compaction contract** (keep_first) trong `agent-harness.md`. spec-guardian thành **read-only thật** (bỏ Bash; caller paste diff). **Loại:** multi-agent song song cho viết code; LLM-judge làm gate cứng (giữ **test là gate**; oracle/spec-guardian chỉ advisory). Chi tiết cơ chế: `agent-harness.md`.

### ADR-024 — Mở rộng harness từ nghiên cứu vòng 2 (chứng minh test ràng buộc · truy vết acceptance · bồi đầu trái · sàn sandbox) · Accepted (user 2026-06-23)

### ADR-029 — Relay outbox→NATS: goroutine in-process · quét-tập-pending theo seq · publish→PubAck→mark · Accepted (user 2026-06-26)

Slice-3 dựng relay phát dòng `outbox` `status='pending'` (slice-2 tích) lên NATS JetStream. **Quyết định:** relay là **một goroutine in-process trong core-api** (KHÔNG binary `cmd/relay` riêng — durability nằm ở dòng đã-commit; tách binary không thêm an toàn mà thêm deployable/conn/lifecycle; ADR-009 minimal + `architecture.md` "core-api phát job qua outbox→NATS"). **Quét cả TẬP pending `ORDER BY seq`** mỗi tick (~1s) — **KHÔNG watermark `seq>cursor`, KHÔNG `SKIP LOCKED`**: `bigserial seq` gán lúc INSERT chứ không lúc COMMIT → tx seq-thấp có thể hiện ra SAU tx seq-cao → watermark bỏ sót vĩnh viễn dòng seq-thấp commit-muộn = **mất event tiền câm** (hazard lớn nhất). Single instance (ADR-009) ⇒ KHÔNG advisory-lock/leader-election. **`publish → await PubAck → mark-published`** (không mark-rồi-publish); `Nats-Msg-Id = outbox.id` (dedup server-side trong duplicate-window); **subject = `event_type` nguyên văn** (relay không tra cứu). **Transient-vs-poison:** lỗi connection → để `pending`, **không đốt `attempts`**; PubAck reject → `attempts++` → `failed` sau `RelayMaxAttempts` (cô lập poison, không chặn head-of-line). Provision 2 stream `ORDERS`(`order.>`,Limits)+`ASSET_JOBS`(`asset_job.>`,WorkQueue); **worker tự sở hữu durable consumer** (AckWait/MaxDeliver→DLQ là knob worker, ADR-007). Crash-recovery: dòng pending bền → restart re-scan; re-publish trùng bị dedup hấp thụ. Relay chỉ ĐỌC dòng đã-commit (`EnqueueOutbox(tx,…)` vẫn là writer duy nhất). **Loại:** binary riêng · watermark/SKIP-LOCKED · advisory-lock · relay hard-code consumer knob.

### ADR-030 — Auth: JWT tự-phát (core-api own login) thay vì Cloudflare Access + RBAC owner/staff · Accepted (user 2026-06-26)
ADR-003 đặt tên `go-chi/jwtauth` nhưng chưa chốt cơ chế; `users` chưa có cột mật khẩu. **Quyết định (user chọn self-issued, KHÔNG dựa Cloudflare Access):** core-api **tự xác thực** — `POST /auth/login` (email+password) → verify `bcrypt` (`golang.org/x/crypto/bcrypt`; x/crypto promote indirect→direct) → phát JWT ký (`go-chi/jwtauth`, *nhất quán* ADR-003) → cookie **httpOnly+Secure+SameSite** (apps/admin là Next SPA → token ngoài tầm JS, chặn XSS-theft); `POST /auth/logout` xoá cookie. `users` thêm `password_hash` (migration `000009_user_credentials`). Middleware verify JWT trên `/admin/*` map `sub`(users.id)→`users.role`→`order.Role`, inject actor `{ByUser=users.id, Role, At=server-clock}`. RBAC middleware owner/staff + `requireOwner` cho cạnh owner-only (reconcile→PAID, →REFUNDED, `PATCH STK`) — **phòng-thủ-tầng** trên guard `internal/order` (guard vẫn authoritative; KHÔNG re-implement RBAC math). **Cổng channel `POST /orders`:** `web` public; `channel=inbox` đòi actor staff/owner (optional-auth mw — inbox đẻ đơn born-PAID không chứng từ; conventions §17/ADR-012 §49). **Guest web genesis** `StatusEvent.ByUser="customer"` (sentinel; ngoại lệ duy nhất của ByUser=users.id). **Chốt trong PR-3e:** seed owner đầu (bootstrap-hash-phải-xoay vs `make seed-owner` CLI), TTL/refresh token, JWT secret qua env, màn login mỏng apps/admin. **Ngoài scope:** auth khách storefront + Pet Tag `Account` (Phase-1, identity khác). **Mở (không chặn):** handshake auth extension MV3. **Loại:** Cloudflare-Access edge-trust (chọn tự-chủ, không phụ thuộc cấu hình CF Access) · session server-side state (chọn JWT stateless cookie).

### ADR-031 — OpenAPI: yaml viết tay = single-source → oapi-codegen `strict-server` + openapi-typescript · Accepted (user 2026-06-26)
ADR-003 bắt OpenAPI là hợp đồng Go↔TS nhưng chưa có spec/`packages/api-client`; `packages/core` Zod là hợp đồng de-facto. **Quyết định:** **một `services/core-api/openapi.yaml` viết tay = nguồn chân lý** → `oapi-codegen` (Go, **mode `strict-server`** — conformance handler↔contract lúc-compile) + `openapi-typescript`/`openapi-fetch` (TS, gói mới `packages/api-client`). **Test parity 4 chiều** (OpenAPI == `internal/order` == `packages/core` Zod == PG enum) chặn nguồn-thứ-hai trôi. Codegen chạy trong verify/CI; `*.gen.go` commit + **path-exclude trong `.golangci.yml`** (output oapi-codegen đụng golangci v2 0-issue). Đổi Review `text`→`body` (+ sync `spec.md §02`). `Order` DTO công khai dạng **nested** (không phải sqlc row phẳng); prop camelCase; tiền `int64`. **Loại:** types-only (chọn strict-server cho drift-protection mạnh hơn) · sinh spec từ code · giữ Zod làm nguồn riêng.

### ADR-032 — HTTP error envelope `{code, messageKey, fields?}` + bảng domain-error→status · Accepted (user 2026-06-26)

### ADR-033 — POST /orders idempotency: HOÃN sang slice sau · Accepted (user 2026-06-26)
Order id sinh per-request → `POST /orders` retry đẻ **đơn trùng + `order.created` trùng** (lỗ thật đường tiền). **Quyết định (user chọn HOÃN):** slice-3 KHÔNG dựng `Idempotency-Key`/bảng dedupe; chấp nhận rủi ro retry-dup tới slice sau (revisit khi storefront-checkout đẻ retry thật; ghi rõ ở plan §Out-of-scope). **Vẫn làm slice-3 (độc lập idempotency):** mã `#LMN-xxxx` = Postgres sequence (3f, migration `000010`) + biên host-trust `paymentProofUrl` (3g). **Caveat end-to-end:** `POST /orders` (web) **nhận** `paymentProofUrl` đã-validate nhưng endpoint presigned-PUT đẻ URL đó HOÃN (ADR-005) → luồng browser→upload→order chưa chạy trọn slice này.

### ADR-034 — Guest order-lookup rate-limit: token-bucket in-process keyed by order CODE (không per-IP, không failure-lockout) · Accepted (user 2026-07-03)
conventions §Bảo mật đòi "token-bucket trong Go + lockout chống dò" cho `/orders/lookup` (PR-P1-n) nhưng không chốt CÁCH. **Quyết định (user 2026-07-03):** limiter **in-process in-memory** (`golang.org/x/time/rate` per-code bucket + lazy TTL-sweep giữ map bounded; reset khi restart — chấp nhận theo ADR-009/014), **KEY theo mã đơn (`code`), KHÔNG per-IP** — vì slice này core-api **chưa có client-IP tin-cậy** (CF-Connecting-IP hoãn tới edge-integration; chi `RealIP` giả-mạo-được). Phân vai defense-in-depth: **CF WAF** chặn sweep per-IP xuyên-mã; **token-bucket per-code** chặn brute-force SĐT của một mã đã biết (0.5/s → không gian SĐT 9-chữ-số bất-khả-thi). **KHÔNG failure-lockout theo mã:** mã đơn tuần-tự/đoán-được (`#LMN-1000,1001,…` từ sequence 000010) + lockout kiểm-tra TRƯỚC khi so SĐT → attacker gửi N-SĐT-sai khoá mã → **chủ đơn thật (đúng mã+SĐT) bị 429 khỏi đơn của chính họ** (DoS; adversarial review wf_4ef2b511). Token-bucket đơn thuần đã đủ chống brute-force nên lockout chỉ thêm rủi ro. **Env-knobs (rate/burst/ttl) HOÃN sang P1-o** (cadence auto-poll chưa chốt, plan §6.6) — hằng-số package tạm, redeploy re-tune. **Loại:** per-IP keying (không có IP tin-cậy in-process) · failure-lockout theo mã (DoS chủ đơn) · store DB-backed (thừa cho single-instance accept-downtime) · Retry-After trên 429 (rò cửa-sổ).

### ADR-035 — Payment-proof upload: presigned POST to dedicated Garage bucket + host-pinned URL + 90-day terminal retention · Accepted (user 2026-07-05)
Quyết định cho Phase 2 P2-c (`docs/plans/phase-2-checkout.md` D-P2-2): ảnh biên lai CK là dữ liệu cá nhân/tài chính phục vụ đối soát thủ công, nên **không proxy qua core-api/Cloudflare Tunnel**. Storefront gọi `POST /checkout/payment-proof-upload` để lấy **presigned POST** S3/Garage ngắn hạn (TTL mặc định 5 phút), rồi browser upload trực tiếp vào bucket riêng `lumin-payment-proofs`. **POST, không PUT:** policy POST mới chặn được `content-length-range` (≤10MB) + exact `Content-Type` (`image/jpeg|png|webp`) ở phía S3; PUT chỉ tin header client khai. Core-api tự sinh key ngẫu nhiên không chứa PII, trả `finalUrl` **host-pinned** từ cấu hình bucket public/CDN; browser chỉ được gửi `finalUrl` đó vào `POST /orders.paymentProofUrl`. Bucket cần CORS cho storefront origin (POST + form headers) và credentials S3 bucket-scoped. **Retention:** giữ ảnh khoảng 90 ngày sau khi đơn vào terminal để đủ xử lý tranh chấp/đối soát, rồi xoá bằng lifecycle/GC bucket (PDPL data-minimization); orphan upload GC là follow-up nếu tích tụ thật. **Loại:** upload qua core-api (tốn tunnel/body, tăng rủi ro log/proxy) · presigned PUT (không enforce size policy) · bucket chung asset public (retention/PII lẫn với model/sprite) · client tự dựng URL cuối (mất host-pin).
